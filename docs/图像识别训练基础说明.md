# NutriMind 图像识别训练基础说明

这份文档是写给第一次接触图像识别训练的同学看的。

如果你现在还不太理解下面这些词：

- 训练
- 模型
- checkpoint
- `best.pth`
- `last.pth`
- `onnx`

没有关系，先看完这份说明就够用了。

---

## 1. 先说最简单的理解

图像识别训练这件事，可以先理解成：

1. 给模型很多带标签的图片
2. 让模型反复学习“这张图像属于什么类别”
3. 在训练过程中不断保存学习结果
4. 最后导出一个可以部署给系统调用的模型文件

所以训练的结果通常不是只有一个文件，而是一整套文件。

---

## 2. 训练过程中通常会得到什么

以你们项目现在的流程来说，训练后通常会得到这些东西：

### 2.1 `best.pth`

这个文件表示：

- 训练过程中“效果最好”的那一次模型存档

它通常用于：

- 后续继续训练
- 导出部署模型
- 保留当前最优结果

如果只留一个训练 checkpoint，通常优先留它。

### 2.2 `last.pth`

这个文件表示：

- 最后一个 epoch 训练结束时的模型存档

它通常用于：

- 训练中断后继续接着训
- 查看训练最后停在哪

但它不一定是效果最好的版本。

### 2.3 `labels.json`

这个文件表示：

- 类别编号和类别名字的对应关系

例如模型输出第 `0` 类、第 `1` 类、第 `2` 类，系统必须知道这些数字到底对应：

- 鸡胸肉
- 米饭
- 鸡蛋

这就是 `labels.json` 的作用。

### 2.4 `metadata.json`

这个文件表示：

- 这个模型的一些说明信息

例如：

- 模型版本
- 输入图片尺寸
- 类别数
- 推荐后端
- 训练样本数

它更像模型的“说明书”。

### 2.5 `.onnx` 文件

这个文件表示：

- 给系统部署和推理使用的“成品模型”

在你们项目里，更重要的是这个文件，因为系统真正上线调用时，通常更关心：

- `.onnx`
- `labels.json`
- `metadata.json`

而不是训练时的 `.pth`。

---

## 3. `best.pth` 和 `last.pth` 到底是什么

你可以把它们理解成“训练过程中的存档”。

比如你写论文、写文档时，电脑里可能会有：

- `最终版.docx`
- `最新修改版.docx`

它们有点像这个意思：

- `best.pth` = 效果最好的那一版
- `last.pth` = 最后保存的那一版

注意：

- `last.pth` 不一定最好
- `best.pth` 不一定是最后一轮

---

## 4. 这两个文件里面一般保存了什么

通常会保存这些内容：

- 模型参数
- 当前训练轮数
- 当前最好的指标
- 有时还会包含优化器状态

这也是为什么它们可以用来“继续训练”。

如果没有这些 checkpoint，而只有导出的 `.onnx`，通常就更适合直接部署，不适合无缝接着之前的训练继续跑。

---

## 5. 在你们项目里，真正部署需要什么

如果目标是“让系统识别图片”，通常更关心下面这几样：

- `food_classifier.onnx` 或类似的 ONNX 文件
- `labels.json`
- `metadata.json`
- 推理服务代码

如果目标是“接着之前的训练继续优化模型”，那才更关心：

- `best.pth`
- `last.pth`

所以你可以这么记：

- **部署看 ONNX**
- **续训看 PTH**

## 7. 能不能从 `.gitignore` 里删掉然后传到 GitHub

### 7.1 从 GitHub 技术限制上说

可以。

因为 GitHub 单文件上限通常是 `100 MB`，而你这两个文件目前都只有约 `42.71 MB`，所以单看大小，是可以上传的。

### 7.2 从项目协作角度说

**不太推荐直接把两个都传上去。**

原因有几个：

- 训练文件会让仓库变重
- 后面如果你们不断更新 checkpoint，Git 历史会越来越臃肿
- 这类文件变化频繁，但对大部分同学不是必须

### 7.3 更推荐的做法

更推荐下面这个策略：

- GitHub 保留代码、文档、配置模板、推理接口
- 训练 checkpoint 单独共享
- 真正部署时优先共享 `onnx + labels.json + metadata.json`

---

## 8. 如果一定要传，传哪个更合适

如果你真的想传一个给队友，优先级建议是：

### 方案 A：最推荐

只传部署相关文件：

- `.onnx`
- `labels.json`
- `metadata.json`

这样队友可以直接跑推理服务。

### 方案 B：如果想让队友“接着你训练”

优先传：

- `best.pth`

因为它通常是当前效果最好的版本。

### 方案 C：什么时候才需要 `last.pth`

只有在下面这种情况才更需要：

- 你想精确保留“最后一次训练停下来的状态”
- 你想从最后轮继续训练，而不是从最佳轮继续训练

对大多数比赛项目来说，`last.pth` 的优先级通常没有 `best.pth` 高。

---

## 9. 对你现在这个项目，我的建议

如果你只是为了让队友继续做这部分，我建议这样分：

### 9.1 如果队友要做推理接入和效果验证

给他这些就够了：

- 推理代码
- `.onnx`
- `labels.json`
- `metadata.json`
- 运行说明

### 9.2 如果队友要继续训练

再额外给他：

- `best.pth`

通常先不用给 `last.pth`。

### 9.3 如果你想保持仓库轻量

那就不要改 `.gitignore`，继续保持：

- `.pth` 不上传 GitHub
- 通过网盘、压缩包、共享盘单独发给队友

这会更稳。

---

## 10. 一句话总结

`best.pth` 和 `last.pth` 都是训练 checkpoint。

其中：

- `best.pth` 更适合保留和续训
- `last.pth` 更像最后一次训练状态备份
- 系统真正部署更看重的是 `.onnx + labels.json + metadata.json`

你现在这两个文件大小都在 `42.71 MB` 左右，**技术上可以传 GitHub，但从项目协作和仓库整洁度来说，不建议两个都传；如果一定要传，优先考虑 `best.pth`，或者更直接地传部署用的 ONNX 文件。**
