目标很明确：

- 让队友知道现在项目已经做到哪一步
- 让队友知道他最适合接手哪一部分

## 1. 先说结论

目前你们项目里的图像识别模块，**整体架构已经搭好了**，但**模型能力本身还没有完全做强**。

所以最推荐的分工是：

换句话说：

**这部分非常适合交给队友做，但前提是他最好围绕你们现有接口继续开发，而不是完全另起炉灶。**

## 2. 当前项目这部分已经完成了什么

### 2.1 Java 服务层已经打通

项目里已经有图像识别的业务入口：

- [VisionController.java](/d:/workspace/NutriMind/ai-service/src/main/java/com/yxw/ai/controller/VisionController.java)

1. 前端上传图片给 `ai-service`

- 业务链路不是空白

## 4. 到底是你继续做，还是交给队友做

我的建议是：

### 4.1 更适合你继续做的部分

- 主项目整体推进

最优方案不是：

- 你全做

而是：

**你负责“系统集成”，他负责“模型能力”。**

这才是最省力、最不容易互相卡住的方式。

不会，这其实是正常做法。

训练数据、模型权重、checkpoint、ONNX 文件本来就经常不会直接放进 GitHub。

比较推荐的方式是：

所以你现在的做法没有问题。

## 6. 队友可不可以做一个独立模型训练，然后导出 API 给你调用

**可以，而且这是非常推荐的做法。**

但是要注意：

### 6.1 可以独立做训练

他可以自己负责：

- 下载或整理数据集
- 跑训练
- 调整模型
- 导出 ONNX 或其他可部署格式
- 搭一个单独的 Python 推理 API

### 6.2 但是不要随便改接口格式

最关键的一点是：

**他的 API 最好兼容你们现在项目已经定义好的返回结构。**

也就是说，不建议让他自己随便设计一套新 JSON。

因为你们现在 Java 端已经按现有字段对接好了。

当前你们已有的返回结构可以参考：

- [schemas.py](/d:/workspace/NutriMind/ai-service/inference/python/app/schemas.py)
- [PythonInferenceResponse.java](/d:/workspace/NutriMind/ai-service/src/main/java/com/yxw/ai/dto/PythonInferenceResponse.java)
- [PythonInferencePrediction.java](/d:/workspace/NutriMind/ai-service/src/main/java/com/yxw/ai/dto/PythonInferencePrediction.java)

### 6.3 推荐的最小兼容接口

接口地址建议继续保持：

- `POST /predict`

请求方式：

- `multipart/form-data`

请求字段：

- `file`：图片文件
- `top_k`：返回多少个候选

推荐返回 JSON 结构如下：

```json
{
  "mode": "classifier",
  "model_version": "food-cls-v2",
  "predictions": [
    {
      "label": "鸡胸肉",
      "canonical_label": "鸡胸肉",
      "confidence": 0.93,
      "match_reason": "模型高置信命中",
      "source": "custom-model",
      "aliases": ["鸡肉", "鸡排"],
      "search_keywords": ["鸡胸肉", "鸡肉"]
    }
  ]
}
```

只要保持这个结构，你这边 Java 服务通常不需要大改。

### 6.4 你这边怎么接他的服务

你这边 `ai-service` 目前已经支持通过配置调用 Python 推理服务，相关配置在：

- [application.properties](/d:/workspace/NutriMind/ai-service/src/main/resources/application.properties)

关键配置是：

```properties
app.vision.engine=python
app.vision.python.base-url=http://localhost:8091
app.vision.python.predict-path=/predict
```

也就是说，如果他把服务部署在：

- `http://localhost:9000`

那你只需要把：

- `APP_VISION_PYTHON_BASE_URL=http://localhost:9000`

改掉就可以联调。

## 10. 你们协作时最需要注意什么

### 10.1 不要让他改你的主项目接口契约

可以改模型。
可以改推理实现。
可以改训练脚本。

但最好不要随便改这些核心约定：

- `/predict` 路径
- `multipart/form-data` 请求方式
- 返回 JSON 主结构

不然你这边 Java 会跟着改，联调成本会变高。

### 10.2 不要把“训练问题”和“系统联调问题”混在一起排查

你们后面联调时，建议按顺序查：

1. 推理服务能不能单独跑
2. `/predict` 单独调通没有
3. Java 能不能调用 Python
4. Java 返回前端有没有问题
5. 食物库匹配结果对不对

不要一上来就把所有问题混在一起。

### 10.3 比赛项目优先“稳定演示”

如果是比赛，不一定非要把模型做到最先进。

更重要的是：

- 能上传图片
- 能返回候选结果
- 能映射到食物库
- 能继续进入后续业务流程

只要演示链路稳定，评委通常就能看到价值。
